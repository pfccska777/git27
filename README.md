### Задание 1. Резервное копирование

### Кейс
Финансовая компания решила увеличить надёжность работы баз данных и их резервного копирования. 

Необходимо описать, какие варианты резервного копирования подходят в случаях: 

1.1. Необходимо восстанавливать данные в полном объёме за предыдущий день.

1.2. Необходимо восстанавливать данные за час до предполагаемой поломки.

1.3.* Возможен ли кейс, когда при поломке базы происходило моментальное переключение на работающую или починенную базу данных.

*Приведите ответ в свободной форме.*

### Решение 1

1.1 Резервное копирование для восстановления данных за предыдущий день

Подходит полный (full) бэкап базы данных, выполняемый один раз в сутки, например, ночью. Такой бэкап сохраняет всю базу данных целиком, что позволяет восстановить всю информацию на начало текущего дня. Этот вариант прост в реализации и обеспечивает полное восстановление данных, но при больших объемах данных может занимать много времени и ресурсов.

1.2 Резервное копирование для восстановления данных за час до поломки

Для точечного восстановления на момент за час до сбоя подходят:

- Инкрементальные (incremental) или дифференциальные (differential) бэкапы, выполняемые периодически в течение дня. Полный бэкап делается раз в сутки, а затем регулярно сделанные инкременты позволяют откатиться к нужному времени с минимальными затратами.
- Журнал транзакций (transaction logs) или бинарные логи, которые фиксируют каждую операцию в базе. Это позволяет восстановить данные вплоть до конкретного времени (например, час назад) с высокой точностью.

1.3 Возможность моментального переключения на работающую или починенную базу при поломке

Да, такой кейс возможен и реализуется через технологии высокой доступности (High Availability, HA):

- Репликация с автоматическим переключением (failover) – активный master-сервер и один или несколько пассивных slave-серверов. При отказе master система автоматически переключается на slave, который становится новым master.
- Кластерные решения и распределённые базы данных с поддержкой failover (например, PostgreSQL с Patroni, MySQL Group Replication).
- Технологии load balancer и heartbeat обеспечивают мониторинг состояния серверов и обеспечивают минимальный простой.

Такой подход позволяет обеспечить практически беспрерывную работу и минимизировать время простоя при отказах.

Итог

- Для восстановления за предыдущий день — полный бэкап раз в сутки.
- Для восстановления до часа — сочетание полного бэкапа с инкрементальными и журнальными логами.
- Для моментального переключения — использование репликации с автоматическим failover и кластерных решений высокой доступности.

### Задание 2. PostgreSQL

2.1. С помощью официальной документации приведите пример команды резервирования данных и восстановления БД (pgdump/pgrestore).

2.1.* Возможно ли автоматизировать этот процесс? Если да, то как?

*Приведите ответ в свободной форме.*

### Решение 2

2.1 Пример команды резервного копирования и восстановления данных в PostgreSQL

- Резервное копирование с помощью pg_dump:


pg_dump -U username -h hostname -F c -b -v -f backup_file.dump dbname

Где:

- -U username — имя пользователя базы.

- -h hostname — хост сервера базы.

- -F c — формат резервной копии custom.

- -b — включает бэкап больших объектов (large objects).

- -v — режим подробного вывода.

- -f backup_file.dump — файл для сохранения бэкапа.

- dbname — имя базы данных.

- Восстановление с помощью pg_restore:


pg_restore -U username -h hostname -d dbname -v backup_file.dump

Где:

- -d dbname — база, куда восстанавливаем.
- Другие параметры аналогичны.

2.1 Автоматизация процесса*

Да, процесс можно автоматизировать. Вот как:

- Планировщик задач (cron в Linux или Task Scheduler в Windows):

Создать скрипт (bash, batch, PowerShell), который выполняет резервное копирование с нужными параметрами. Затем настроить расписание выполнения скрипта, например, раз в сутки или чаще.

- Скрипты с логированием и уведомлениями:

Автоматически сохранять логи выполнения, отправлять письма или оповещения при успехе или ошибках.

- Использование инструментов управления резервным копированием:

Например, pgBackRest или Barman, которые предоставляют удобное управление, сжатие, шифрование и автоматизацию.

Кратко

- pg_dump и pg_restore — стандартные инструменты для бэкапа и восстановления PostgreSQL.
- Автоматизацию реализуют с помощью скриптов и планировщиков задач.
- Для более продвинутого управления — специализированные инструменты резервного копирования.

### Задание 3. MySQL

3.1. С помощью официальной документации приведите пример команды инкрементного резервного копирования базы данных MySQL. 

3.1.* В каких случаях использование реплики будет давать преимущество по сравнению с обычным резервным копированием?

*Приведите ответ в свободной форме.*

### Решение 3

3.1 Пример команды инкрементного резервного копирования MySQL

Для инкрементного резервного копирования в MySQL обычно используется механизм бинарных логов (binary logs), который фиксирует все изменения данных. Пример:

1. Включить бинарные логи в my.cnf (если ещё не включены):

[mysqld]
log-bin=mysql-bin
server-id=1

2. Сделать полное резервное копирование с помощью mysqldump:

mysqldump -u username -p --single-transaction --flush-logs --master-data=2 dbname > full_backup.sql

3. Для инкрементного бэкапа сохранить бинарные логи, созданные после полного бэкапа:

mysqlbinlog mysql-bin.00000X > incremental_backup.sql

Где mysql-bin.00000X — файлы бинарного лога, созданные после полного бэкапа.

Таким образом инкременты — это серии бинарных логов, которые можно применить для восстановления изменений после полного бэкапа.

3.1 Преимущества использования реплики по сравнению с обычным резервным копированием*

- Минимизация нагрузки на основную БД — реплика занимает на себя операции резервного копирования, снижай нагрузку и не блокирует основную базу.

- Быстрое восстановление и отказоустойчивость — реплика может служить как горячий резерв, автоматически подменяя основной сервер при сбое.

- Возможность создания консистентного бэкапа — на реплике можно делать снимок данных без прерывания работы основной БД, что особенно важно при больших объемах данных.

- Синхронизация и анализ данных — реплика позволяет проводить аналитические запросы и тесты без влияния на производительность основной БД.

Таким образом, репликация — это не только способ повышения отказоустойчивости, но и эффективный способ организации инкрементного резервного копирования с минимальной нагрузкой.
